## Confirmed Hardened Areas

- Server startup now requires an explicit egress posture: either `--allow` rules
  or `--allow-open-egress`.
- Client and server reject plaintext OIDC/tunnel URLs by default, with explicit
  insecure development overrides.
- Token validation now covers issuer, signature, expiration, audience, non-empty
  subject, future `iat`, and `nbf` admission/refresh policy.
- Refresh continuity is pinned to the original token subject.
- Admission controls exist for global concurrency, per-user concurrency,
  per-user open rate, and outbound dial timeout.
- WebSocket control frames have a size cap even though binary tunnel frames are
  unbounded for streaming.
- The POSIX client socket/cache directory checks reject writable leaf
  directories and unsafe ancestors, and stale socket removal is type checked.

## Workstream 1: Bound OIDC, JWKS, and Pre-Auth Work

Status: Done in `a889e12b3444fc6f4251fa44a3b65fb79e52c296`
(`Pre-auth rate, and oversized data, limiting. Bound OIDC/JWKS calls.`),
merged from local `main` on 2026-04-30. Additional local regression coverage
was added after for JWKS fetch timeout and authorization-code exchange
 timeout.

### Risk

The server creates the token validator with `context.Background()` and
`http.DefaultClient`, and the validator uses the supplied client for OIDC
discovery and JWKS fetches. A slow or stalled issuer can therefore hang startup
or hold request goroutines during token validation. The managed client also
uses the default HTTP client unless tests inject one.

Invalid or oversized bearer-token requests are rejected only after the request
has reached the token-validation path. That leaves room for unauthenticated
resource exhaustion through large headers, expensive JWT parsing, or repeated
unknown-`kid` JWKS refresh attempts.

### Completed changes

- Added `internal/authhttp.NewBoundedClient`, with dial, TLS-handshake,
  response-header, idle, and overall HTTP timeouts for OIDC discovery, JWKS,
  refresh, and token endpoint calls.
- The server now uses the bounded auth client and wraps startup OIDC discovery
  in a `30s` context so a stalled issuer fails closed during startup.
- The managed client now defaults to the same bounded auth client when tests or
  callers do not inject one.
- JWKS validation still receives the request context through
  `ValidateAccessToken`, while the underlying remote-key HTTP client now also
  has its own timeout.
- Bearer tokens are capped at `8 KiB`, and the full `Authorization` header is
  capped at `8 KiB + 64 bytes`, before the validator is invoked. Oversized
  credentials reuse the fixed `invalid header` response.
- `http.Server.MaxHeaderBytes` is lowered to `16 KiB` to bound anonymous
  request-header memory before handler code runs.
- Added an optional pre-auth limiter configured by `--preauth-rate` and
  `--preauth-burst` (or `PREAUTH_RATE` and `PREAUTH_BURST`). It runs before
  token parsing on all protected routes and keys on the TCP peer, or on the
  leftmost `X-Forwarded-For` entry when plaintext reverse-proxy mode is
  explicitly enabled.
- README now documents the new auth bounds, pre-auth flags, and reverse-proxy
  `X-Forwarded-For` caveats.

### Completed tests

- `TestNewJWTTokenValidatorDiscoveryTimesOut` covers stalled server-side OIDC
  discovery with a bounded HTTP client.
- `TestNewJWTTokenValidatorRespectsContextDeadline` covers the startup
  discovery context deadline.
- `TestJWTTokenValidatorJWKSFetchTimesOut` covers stalled JWKS fetches during
  `ValidateAccessToken`.
- `TestNewAuthTokenSourceDefaultsToBoundedClient`,
  `TestManagedOIDCTokenSourceDiscoveryTimesOut`, and
  `TestManagedOIDCTokenSourceCodeExchangeTimesOut` cover the managed-client
  bounded-client default, discovery timeout, and authorization-code exchange
  timeout.
- `TestValidateRequestTokenRejectsOversizedAuthorizationHeader`,
  `TestValidateRequestTokenRejectsOversizedBearerToken`, and
  `TestValidateRequestTokenAcceptsTokenAtBoundary` cover pre-validator
  credential size bounds.
- `TestHandlerPreAuthDeniedBeforeValidator` and
  `TestHandlerPreAuthGatesAllProtectedPaths` cover pre-auth rate limiting
  before validator/JWKS work across the protected route surface.
- `TestHTTPServerMaxHeaderBytesEnforced` covers server-level oversized-header
  rejection before the handler runs.
- Pre-auth config parsing tests cover flags, environment variables, burst
  defaulting, and invalid combinations.

## Workstream 2: Keep Longevity Enforcement Independent of Peer Reads

Status: Done in `bfd04063b3f78f460f99dd0ac6973a11fdcabb7a`
(`Tunnel expiry divorced from courtesy message logic`), merged from local
`main` on 2026-04-27.

### Risk

`manageTunnelLongevity` sends warnings and token refresh responses with
`conn.SendControl`, which writes using the tunnel context and no per-write
deadline. If a peer stops reading, a warning or refresh response can block the
longevity goroutine. While blocked, that same goroutine cannot observe the
token-expiry or max-duration timer, so an authenticated but malicious client
could delay forced tunnel closure.

The disconnect path already uses `SendControlTimeout`; the same bounded-write
property should apply to all control messages that are sent from the enforcement
goroutine.

### Completed changes

- Peer-visible control frames now go through a buffered channel drained by a
  dedicated writer goroutine.
- The writer sends with `SendControlTimeout(2 * time.Second)`, so slow peer
  reads are bounded at the WebSocket write boundary.
- The deadline-critical select in `manageTunnelLongevity` now enqueues warning,
  disconnect, token accepted, and token rejected messages instead of writing
  them inline.
- The implementation logs `control_send_dropped` when the best-effort queue is
  full and treats cancellation as the authoritative enforcement action.

### Completed tests

- `TestExpiryCancellationFiresWhenPeerStopsReading` covers the original P1:
  a peer that stops reading control frames cannot prevent token-expiry
  cancellation from firing.
- Existing refresh success, same-expiry, expiry-grace, and max-duration tests
  continue to cover the normal longevity paths.

## Workstream 3: Validate Existing Secret Files and Key Material

Status: Done in `2619df9dc2edb08baa77da68e41f2c9cc382386a`
(`local-credential and private-key protection`), merged from local `main` on
2026-04-28.

### Risk

New OIDC token-cache writes use `0o600`, but `loadCache` reads an existing
cache file without validating the file's own owner, type, symlink status, or
mode. A legacy or manually modified `tokens.json` with group/world-readable
mode can continue to leak access and refresh tokens, especially because an
existing non-writable `0o755` config directory is accepted.

The same class of issue exists on the server side for ACME cache directories
and TLS private key files. `checkACMECacheDir` creates missing directories with
`0o700`, but it does not reject an existing permissive directory before
`autocert.DirCache` stores certificate private keys there. TLS key files are
passed to `ServeTLS` without a local permission check.

### Completed changes

- POSIX-safe filesystem helpers moved into `internal/safefs` so client and
  server paths share one audited implementation.
- Existing OIDC token caches are validated with `safefs.EnsurePrivateFile`
  before being read. POSIX requires a regular, non-symlink file owned by the
  current user with no group/world mode bits.
- Symlinked token cache files are rejected rather than followed.
- ACME cache directories are validated with `safefs.EnsurePrivateDir` before
  `autocert.DirCache` can store certificate private keys there, and startup
  probes writability with a temporary file.
- File-based TLS mode validates the private key with
  `safefs.EnsureUnreadableByOthers` before `ServeTLS`.
- Windows keeps a separate implementation that documents the ACL limitation
  instead of pretending POSIX mode-bit checks apply.

### Completed tests

- Existing `tokens.json` with `0o644` is rejected before use.
- Existing `tokens.json` with `0o600` is accepted.
- Symlinked token cache is rejected.
- Token cache owned by another uid is rejected where the test environment can
  exercise ownership.
- Existing ACME cache directory with group/world writable mode is rejected.
- Existing ACME cache directory with safe mode is accepted.
- TLS private key file with group/world-readable mode is rejected on POSIX.
- Windows tests document the ACL behavior instead of assuming POSIX semantics.

## Workstream 4: Guard Hostname Allow Rules Against Sensitive Resolved IPs

Status: Done in `37a0f9a2a6b4077234c3d09b736bc3722b246d1b`
(`IP blocklist functionality in server`), merged from local `main` on
2026-05-02.

### Risk

Hostname allow rules authorize based on the requested FQDN, while CIDR rules
authorize based on the resolved IP. This is clear and documented, but it means
a permitted hostname pattern can reach a dangerous resolved address if DNS is
wrong, compromised, wildcarded too broadly, or controlled by an attacker within
the allowed namespace.

The most sensitive cases are server-local loopback, link-local, and metadata
endpoints. A rule such as `*.internal:22` should not accidentally authorize
`169.254.169.254` or `127.0.0.1` just because the hostname matched.

### Completed changes

- Added `IPBlocklist`, a resolved-IP deny-list that runs after the allowlist.
  Deny wins regardless of whether access was granted by hostname rule, CIDR/IP
  rule, or `--allow-open-egress`.
- The default deny-list covers loopback, IPv4/IPv6 link-local including IMDS
  `169.254.169.254`, unspecified, and multicast ranges. RFC1918, CGNAT, and
  IPv6 ULA ranges stay usable by default.
- Added `--ip-block` / `IP_BLOCK` to replace the default list with explicit
  operator-chosen ranges, and `--no-ip-block` / `NO_IP_BLOCK=true` as the loud
  opt-out when protected destinations are genuinely needed.
- Kept `--allow-open-egress` subject to the default deny-list. Fully open
  posture now requires both `--allow-open-egress` and `--no-ip-block`.
- Added `socks_connect_denied_ip_blocked` logs with `resolved_ip` and stable
  `reason` labels (`loopback`, `link_local_ipv4`, `link_local_ipv6`,
  `unspecified`, or `multicast`).
- README now documents the deny-list defaults, custom list behavior, opt-out,
  and the interaction with open-egress mode.

### Completed tests

- `TestIPBlocklistBlocks` covers loopback, link-local, IMDS, unspecified,
  multicast, IPv4-mapped loopback, and allowed RFC1918/CGNAT/ULA ranges.
- `TestObservedSOCKSRuleSetDeniesAllowedHostnameResolvingIntoBlocklist` covers
  hostname allow plus resolved loopback denial.
- `TestObservedSOCKSRuleSetDeniesIMDSWhenAllowedByHostname` covers hostname
  allow plus resolved metadata IP denial.
- `TestObservedSOCKSRuleSetDeniesLoopbackEvenWithExplicitAllowRule` documents
  the chosen override policy: explicit `--allow` alone does not bypass the
  default deny-list.
- `TestObservedSOCKSRuleSetAcceptsLoopbackWhenIPBlockEmpty` and
  `TestObservedSOCKSRuleSetAcceptsLoopbackWithCustomIPBlockExcludingIt` cover
  the explicit override paths.
- `TestObservedSOCKSRuleSetAcceptsHostnameMatchResolvingToRFC1918` covers
  existing normal hostname behavior for private services.
- `TestObservedSOCKSRuleSetDeniesOpenEgressResolvingToIMDS` and
  `TestObservedSOCKSRuleSetAcceptsOpenEgressWithEmptyIPBlock` cover the
  open-egress policy decision.
- Server config tests cover default population, `--ip-block`, `IP_BLOCK`,
  `--no-ip-block`, `NO_IP_BLOCK`, invalid entries, and mutually exclusive
  combinations.

## Workstream 5: Make Audit Checks Enforced, Not Optional

### Risk

The codebase is small and security-sensitive, so formatting and race checks are

CI still does not enforce formatting or race execution.

### Completed changes

## Workstream 6: Lower-Priority Security Decisions

Status: Done. Items 1 and 2 were implemented in
`6d2bf46844e6d411d2c066f34f4eed13f0c62d56` (`remove client --access-token
flag, warn if server allows unbounded connections`), merged from local `main`
on 2026-05-02. Items 3 and 4 were explicitly deferred after review; rationale
is recorded below so the decision remains auditable.

- **Done — Item 1: warn on unbounded tunnel lifetime.** Server logs
  `connection_lifetime_unbounded` at startup when
  `--no-connection-token-expiry` is set and `--max-connection-duration` is
  `0`. Hard rejection rejected as too disruptive for operators who run long
  tunnels behind their own external timeouts; warning matches the existing
  posture-warning pattern (`egress_mode_open`, `ip_block_disabled`).
- **Done — Item 2: remove `--access-token` flag.** Originally planned as a
  CLI-vs-env source warning; on review the cleaner fix was to remove the
  command-line flag entirely. Bearer tokens for manual mode now come only
  via the `ACCESS_TOKEN` environment variable, eliminating the
  argv/shell-history exposure at source. README updated with secrets-manager
  command-substitution examples that keep the literal token out of history.
- **Deferred — Item 3: HTTPS enforcement in `NewJWTTokenValidator`.** The
  config-time check at `server/server.go` already gates issuer scheme.
  Moving the check into the validator is defense in depth that pays off
  only when a second caller of `NewJWTTokenValidator` exists; today there
  is one. It would also force a `WithInsecureIssuer(true)` override hook
  for the existing tests (`server_test.go` HTTP-issuer cases and
  `client/keycloak_e2e_test.go`) — net new surface area for a hypothetical
  bug. Park until a second caller materialises.
- **Deferred — Item 4: OS keychain for refresh tokens.** Three-OS
  complexity (Keychain / DPAPI / Secret Service), cgo on macOS, and real
  test-infrastructure cost. The plaintext token cache with `safefs`
  mode-bit validation is already the standard mitigation for this class of
  threat, and README's "Token cache at rest" section makes the
  plaintext-on-disk threat model explicit. That satisfies the plan's
  "threat model should remain explicit" goal without committing to the
  platform work.
